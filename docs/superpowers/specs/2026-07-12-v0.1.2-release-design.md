# v0.1.2 Release Design

## 背景

`v0.1.1` 发布后，项目完成了三项面向稳定性和交付质量的改进：

- 将 stream read timeout 覆盖改为确定性测试，消除全量 race 测试中的时序不稳定。
- 增加无需真实 Azure 凭据的本地 Azure OpenAI fake upstream smoke，并纳入 CI
  `release-check`。
- 通过 `.gitattributes` 强制 `*.sh` 使用 LF，并增加 LF/CRLF 自动检查，解决 Windows Git
  checkout 阻断 WSL Bash 的问题。

这些变更不扩大外部 API 契约，适合作为 patch release `v0.1.2` 发布。

## 目标

- 发布版本 `0.1.2`，Git tag 为 `v0.1.2`。
- 在 `CHANGELOG.md` 准确记录用户可感知的验证与开发体验改进。
- 同步 `docs/release.md` 中已扩展的 `release-check` 组成。
- 在 WSL `Ubuntu-24.04` 中以 `VERSION=0.1.2` 完成发布前验证。
- 只在发布提交通过验证并同步远端后创建 tag。

## 非目标

- 不增加新的网关 API、provider 行为或配置字段。
- 不构建或推送 Docker registry 镜像。
- 不创建 GitHub/Gitee Release artifact。
- 不自动生成 changelog。
- 不修改 `v0.1.0` 或 `v0.1.1` 的历史条目。

## Changelog

保留空的 `Unreleased`，在其后新增：

```markdown
## 0.1.2 - 2026-07-12

### Added

- Credential-free local Azure OpenAI smoke coverage for chat completions, streaming chat completions, and embeddings, included in release checks.
- Repository-enforced LF line endings and automated LF/CRLF checks for shell scripts.

### Fixed

- Stabilized stream read timeout race coverage by replacing timing-dependent duplicate tests with deterministic shared HTTP stream tests.
```

Changelog 使用项目现有英文风格，不记录设计/计划文档、worktree ignore 等内部过程提交。

## 发布文档同步

更新 `docs/release.md` 的 Preflight 列表，使其与当前 `Makefile` 一致，至少包含：

- `make verify`，其中包括 shell line ending 检查、普通测试、race 和 vet。
- `make check-config`。
- `make check-config-examples`。
- `make build VERSION=<version>`。
- `make smoke`。
- `make smoke-rate-limit`。
- `make smoke-azure`。
- `make smoke-deepseek-skip`。

示例版本从 `0.1.0` 更新为 `0.1.2`，包括 preflight、build、Docker build 和 tag 命令；不改变发布
流程语义。

## Task 156

新增 `tasks/156-v0.1.2-release.md`，记录：

- 发布范围。
- changelog 和 release 文档同步。
- `make release-check VERSION=0.1.2` 验收。
- 发布提交和 tag 名称。

Task 在发布提交中标记 `Done`；tag 推送结果在最终交付消息中记录，不向 Task 追加 tag 后提交，避免
tag 不再指向最终发布提交。

## 发布验证

在应用 `.gitattributes` 后的干净 worktree 中运行：

```bash
make release-check VERSION=0.1.2
```

必须确认：

- `line-endings-ok` 和 `line-endings-test-ok`。
- 普通测试、race、vet 全部通过。
- config、自检和 build 通过。
- fake、rate limit、Azure smoke 通过。
- DeepSeek 无 key 路径安全跳过。
- 构建产物版本通过 `go version -m` 或启动 `/version` 验证为 `0.1.2`；优先使用短生命周期本地
  启动并请求 `/version`，以同时验证 ldflags 和 runtime endpoint。

发布验证不需要真实 provider key，不访问 Azure 或 DeepSeek。

## 提交、同步与 Tag

发布顺序固定为：

1. 修改 changelog、release 文档和 Task 156。
2. 执行完整发布验证。
3. 提交：`release: prepare v0.1.2`。
4. 推送发布提交到 `origin/master`。
5. 确认本地 `master` 与 `origin/master` 指向同一提交。
6. 创建 annotated tag：

   ```bash
   git tag -a v0.1.2 -m "Release v0.1.2"
   ```

7. 推送 tag：

   ```bash
   git push origin v0.1.2
   ```

8. 确认 `v0.1.2^{commit}` 与发布提交一致。

如果发布提交推送、远端同步或最终验证失败，不得创建 tag。若 tag 已在本地创建但尚未推送且发现问题，
停止并报告，不自动删除或移动 tag。

## 风险与控制

- Tag 是对外发布标识，必须在用户明确批准实际发布动作后创建和推送。设计/计划批准不等于 tag
  推送批准。
- `make release-check` 会执行 `gofmt -w`；提交前确认没有无关 Go 变更。
- WSL 偶发 `WSL_E_DISTRO_NOT_FOUND` 时，只允许在确认测试尚未启动后重试同一命令。
- 发布文档可能落后于 Makefile；本任务逐项对照 `release-check` 依赖，避免遗漏 Azure smoke 或
  line-ending 检查。
