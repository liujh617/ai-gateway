# Agent Audit JSONL Design

## Goal

Add an explicitly enabled local research audit mode for studying how third-party agents build prompts, carry context, call models, and receive responses through the gateway.

The audit mode records full request and response bodies to a local JSONL file so the operator can inspect agent context and prompt evolution across requests.

## Non-Goals

- This is not a production compliance audit feature.
- This is not enabled by default.
- This does not send audit data to a remote service.
- This does not record gateway API keys, upstream API keys, or `Authorization` headers.
- This first version does not provide a web UI, database, search index, redaction engine, or retention policy.

## Configuration

Add an `audit` section to JSON config:

```json
{
  "audit": {
    "enabled": true,
    "path": "audit/agent-trace.jsonl"
  }
}
```

Defaults:

- `audit.enabled`: `false`
- `audit.path`: `audit/agent-trace.jsonl`

Environment overrides:

- `GATEWAY_AUDIT_ENABLED`
  - Accepted true values: `1`, `true`, `yes`, `on`
  - Accepted false values: `0`, `false`, `no`, `off`
  - Any other non-empty value is a config validation error.
- `GATEWAY_AUDIT_PATH`
  - Overrides `audit.path`.
  - Must be non-empty and must not contain leading or trailing whitespace.

If audit is disabled, the gateway must not create or write the audit file.

If audit is enabled, `audit.path` is required after defaults and environment overrides are applied.

## Trace Identity

Audit events use:

- `request_id`: the gateway request id already generated by request-id middleware.
- `trace_id`: the value of request header `X-Agent-Trace-Id` when present and non-empty after trimming; otherwise the request id.

The audit event must not include arbitrary request headers. Only the derived `trace_id` is recorded.

## Event Format

Each audit record is one JSON object written as one line to the configured JSONL file.

Common fields:

```json
{
  "timestamp": "2026-07-11T01:23:45.000000000Z",
  "event": "request",
  "request_id": "req_...",
  "trace_id": "agent-session-...",
  "path": "/v1/chat/completions",
  "client": "default",
  "external_model": "gpt-4o-mini",
  "provider": "openai",
  "upstream_model": "gpt-4o-mini",
  "status": 200,
  "duration_ms": 12,
  "body": {}
}
```

Field rules:

- `timestamp`: UTC RFC3339Nano.
- `event`: one of `request`, `response`, `stream_chunk`, `stream_done`, `error`.
- `request_id`: gateway request id.
- `trace_id`: derived trace id.
- `path`: normalized route path.
- `client`: non-sensitive gateway client name.
- `external_model`: client-visible model when known.
- `provider`: selected provider when known.
- `upstream_model`: upstream model when known.
- `status`: HTTP status code when known. For request events before a status exists, omit the field.
- `duration_ms`: request duration when known. For request events before completion, omit the field.
- `body`: raw JSON payload represented as JSON, not as an escaped string, when the payload is valid JSON.

## Covered Endpoints

First version covers:

- `POST /v1/chat/completions`
- `POST /v1/embeddings`

`GET /v1/models`, runtime probes, `/metrics`, unknown routes, and method-not-allowed responses are not content-bearing agent interactions and are out of scope for first version.

## Chat Completions Behavior

For non-streaming chat completions:

1. After request JSON is decoded and validated enough to know `model`, write a `request` event with the full client request body.
2. After provider response is ready and before returning to the client, write a `response` event with the full response body as returned by the gateway.
3. If the request fails after body decode, write an `error` event with status and OpenAI-compatible error body.

For streaming chat completions:

1. After request JSON is decoded and validated enough to know `model`, write a `request` event with the full client request body.
2. For every upstream chunk successfully sent to the client, write a `stream_chunk` event with that chunk body.
3. When `[DONE]` is sent, write a `stream_done` event.
4. If the stream ends due to cancellation or timeout after the SSE response has started, write an `error` event with the error category but do not write prompt or response headers.

The first version records chunks rather than trying to reconstruct a final assistant message. Reconstruction can be done offline from JSONL.

## Embeddings Behavior

For embeddings:

1. After request JSON is decoded and validated enough to know `model`, write a `request` event with the full client request body.
2. After provider response is ready and before returning to the client, write a `response` event with the full response body as returned by the gateway.
3. If the request fails after body decode, write an `error` event with status and OpenAI-compatible error body.

Embedding vectors may be large. This is acceptable for the research mode because it is explicitly enabled and local-only.

## Storage

Add an `internal/audit` package with:

- `Config`
- `Event`
- `Recorder`
- `NewJSONLRecorder(path string) (*Recorder, error)`
- `Record(ctx context.Context, event Event)`
- `Close() error`

Behavior:

- Append to the file.
- Create parent directories when needed.
- Serialize writes with a mutex so concurrent requests do not interleave JSONL lines.
- Best effort for per-request audit writes: a write failure should be logged but must not fail the model request.
- Startup failure to open the configured audit file should fail gateway startup or config check, because enabled audit means the operator explicitly asked for local recording.

## Integration

API server receives an optional audit recorder through `api.Options`.

Config layer loads:

- `audit.enabled`
- `audit.path`

Command layer:

- Builds the audit recorder when enabled.
- Passes it to `api.NewServer`.
- Closes it during server shutdown.
- Includes non-sensitive audit summary in config check output:
  - `audit_enabled`
  - `audit_path`

The audit package must not depend on `internal/api`, `internal/provider`, or HTTP server internals.

## Security Notes

This feature intentionally records full request and response bodies. It can include prompts, completions, tool schemas, user data, retrieved context, embedding inputs, and embedding vectors.

Safety constraints:

- Default disabled.
- Documentation must clearly mark it as local research mode.
- Never record `Authorization` header.
- Never record gateway API keys or upstream provider API keys.
- Do not add audit output to access logs or metrics.
- Example config must use obvious local paths and placeholder keys.

## Testing

Unit and API tests should cover:

- Config defaults keep audit disabled.
- Config file can enable audit and set path.
- Environment variables override enabled/path.
- Invalid `GATEWAY_AUDIT_ENABLED` value fails config validation.
- Audit disabled does not create a file.
- Non-streaming chat writes `request` and `response` events with full JSON bodies.
- Embeddings writes `request` and `response` events with full JSON bodies.
- Streaming chat writes `request`, `stream_chunk`, and `stream_done`.
- `X-Agent-Trace-Id` controls `trace_id`; missing header falls back to request id.
- Audit write failure after startup does not fail the model request.

Verification commands:

```powershell
wsl.exe -d Ubuntu-24.04 --cd /mnt/e/code/open-ai-gateway -- bash -lc "go test ./internal/audit ./internal/config ./internal/api"
wsl.exe -d Ubuntu-24.04 --cd /mnt/e/code/open-ai-gateway -- bash -lc "make verify"
```
